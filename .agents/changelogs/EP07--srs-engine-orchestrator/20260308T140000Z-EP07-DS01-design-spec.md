## 1. Feature Overview

Introduce `SrsEngine` — a class that wires EP02–EP06 modules together behind a single, stable public API. Callers (future backend, demo scripts, BDD tests) interact only with `SrsEngine`; internal module details remain hidden.

Two methods:

- `composeBatch(wordStates, options?)` — produces a `Batch` for the next quiz session
- `processAnswers(answers, wordStates)` — applies all SRS rules and returns updated `WordState[]`

---

## 2. Core Requirements

| Requirement                         | Decision                                             | Rationale                                       |
| ----------------------------------- | ---------------------------------------------------- | ----------------------------------------------- |
| Config validation                   | Throw descriptive errors on construction             | Fail fast; no silent misconfiguration           |
| `composeBatch` output               | Delegates to `batch.ts` after filtering              | Keeps composition logic in one place            |
| Shelved word filtering              | Filter before `getEligibleWords`                     | Shelved words must never appear in a batch      |
| `processAnswers` return shape       | New immutable `WordState[]` (same type)              | Consistent with existing pure-function pattern  |
| FSRS gate                           | Only for words in `srsM2_review` pre-answer          | Avoids scheduling freshly promoted words        |
| Foundational wrong rule gate        | Only for `foundational` + `learning` pre-answer      | Avoids double-penalising srsM2_review demotions |
| Shelve duration                     | 24h constant (`SHELVE_DURATION_MS`)                  | Not user-configurable; fixed policy             |
| `batchesSinceLastProgress` tracking | Learning-phase words only, reset on mastery progress | srsM2_review words use FSRS intervals instead   |

## 4. `processAnswers` Composition Order

Per answered word (using **pre-answer** `state` for all gates):

1. `updateMastery(state, isCorrect, config)` → `next`
2. If `state.phase === 'srsM2_review'`: apply `FsrsScheduler.scheduleReview` → merge `fsrsState` into `next`
3. If `state.category === 'foundational' && state.phase === 'learning'`:
   - Wrong: `applyFoundationalWrongRule(next, config)` → `next`
   - Correct: reset `consecutiveWrongCount → 0` on `next`
4. If `state.phase === 'learning'`: update `batchesSinceLastProgress` (reset to 0 on progress, increment otherwise)

After all words processed:

5. `detectStuckWords(updated, config)` → shelve `toShelve` words with `SHELVE_DURATION_MS`

---

## 5. `composeBatch` Logic

```
unshelved = wordStates.filter(not isShelved)
{ active, newSlots, eligible } = getEligibleWords(unshelved, config)
forBatch = [...active, ...eligible.slice(0, newSlots)]
return composeBatch(forBatch, config, options)
```

| `maxShelved`                    | > 0         |

## Notes

- **`processAnswers` is pure** — no side effects; same immutable-update pattern as all other engine functions
- **FSRS gate on pre-answer phase** — critical: `state.phase` (not `next.phase`) determines whether FSRS runs; otherwise a word graduating in this call would also get scheduled
- **Foundational wrong rule gate on pre-answer phase** — same reason: a word demoted from srsM2_review in this call must not also get the consecutive wrong penalty
- **`batchesSinceLastProgress` only for learning phase** — srsM2_review words use FSRS intervals for scheduling; tracking batches for them has no meaning
- **No CODEMAP for `__tests__/`** — per RULES.md; integration test file is self-documenting via file-level doc comment
