## Correctness Bugs

### BUG-1 ‚Äî `recheckReentered` words silently retired before second-chance attempt ‚öÝÔ∏è HIGH

**File:** `packages/srs-engine-v2/src/engine/adaptive-session.ts:81`

`advanceAdaptiveSession` passes only `recheckPending` as the exempt set to `nextActivePool`, not `recheckReentered`. A word that fails its first recheck is moved from `recheckPending` ‚Üí `recheckReentered` by `processRecheckResult` ‚Äî but on the same `advanceAdaptiveSession` call, `nextActivePool` no longer considers it exempt. Since the first-recheck path suppresses streak/mastery changes (only increments `seen`/`correct`), the word's mastery remains at threshold. `isMastered` returns true and the word is filtered out of `active`. The second-chance attempt the design promises never happens.

**Fix:** Pass `new Set([...recheckPending, ...recheckReentered])` as `recheckExempt` in `advanceAdaptiveSession`.

```ts
// adaptive-session.ts
const { active, queue } = nextActivePool(
  state.active,
  state.queue,
  config.wordsPerBatch,
  runState,
  config.masteryThreshold,
  new Set([...recheckPending, ...recheckReentered]),  // was: recheckPending only
);
```

### BUG-2 ‚Äî `recheckReentered` deletion logic inverted ‚öÝÔ∏è HIGH

**File:** `packages/srs-engine-v2/src/engine/session.ts:63`

In `processRecheckResult`, `nextReentered.delete(wordId)` fires only when `!isMastered(wordState, masteryThreshold)` is true ‚Äî i.e. when the word is *still failing*. Words that master on their second recheck stay in `recheckReentered` permanently (state leak); words that keep failing are prematurely evicted from the set (losing their special recheck status).

**Fix:** Invert the condition ‚Äî remove from `recheckReentered` when the word *is* mastered (i.e. graduated):

```ts
// session.ts
if (nextReentered.has(wordId)) {
  const wordState = nextState.get(wordId);
  if (wordState && isMastered(wordState, masteryThreshold)) {  // was: !isMastered
    nextReentered.delete(wordId);
  }
}
```

### BUG-3 ‚Äî Resumed completed session shows blank 0/0 results screen

**File:** `apps/srs-demo/src/App.vue:196`

`onResume` transitions directly to `screen='results'` when `active.length === 0 && queue.length === 0`, bypassing `finishBatchAndTransition`. `batchScore` stays at `{correct: 0, total: 0}` and `summary` stays `[]`. The user sees an empty scoreboard with no word data.

**Fix:** When resuming into a completed session, either render a dedicated "session complete" screen, or populate `batchScore`/`summary` from the restored `sessionState.runState` before setting `screen='results'`.

### BUG-4 ‚Äî `debugSentenceEligibility: true` bypasses all gates in tests ~~(spacing gate off-by-one)~~

**File:** `packages/srs-engine-v2/demo/config.ts:14`

**Corrected finding** (original RV01 analysis was wrong about `<=`): The `<=` in the spacing gate is intentional ‚Äî `sentenceBatchGap=1` means "skip at least 1 batch", so `gap > 1` is required. The actual bug is `debugSentenceEligibility: true` in `config.ts`, which short-circuits the entire `mockSentenceCorpus.filter(...)` block in `resolveEligibleContexts`, returning all sentences unfiltered. Since tests import `resolveEligibleContexts` directly and it reads from `LEARNING_CONFIG`, all gate tests were silently no-ops.

**Fix:** Set `debugSentenceEligibility: false` (the debug flag should be off by default).

**‚úÖ Fixed** ‚Äî `demo/config.ts:14` set to `false`. All 191 tests now green.

### BUG-5 ‚Äî `runInteractive` returns retry-inflated `total`

**File:** `packages/srs-engine-v2/demo/learning-io.ts:212`

`runInteractive` returns `total: count` where `count` increments for every question served including retries. The caller `runAdaptiveLoop` accumulates `totalQuestions += total`, so the end-of-session score denominator is inflated. The in-batch console log uses `state.initialCount` (correct), but the session summary uses the inflated count ‚Äî producing mismatched denominators (e.g. "4 / 7" instead of "4 / 5").

**Fix:** Return `total: state.initialCount` instead of `total: count`, or add a separate `served: count` field for diagnostics.

### BUG-6 ‚Äî Retry counter incremented before cache-miss guard (PLAUSIBLE)

**File:** `packages/srs-engine-v2/src/engine/batch-queue.ts:119`

In `submitBatchResult`, `nextBatchRetryCounts.set(id, batchRetries + 1)` fires unconditionally before checking `if (cached)`. If `questionCache.get(id)` returns `undefined` (a synthetic result submitted for a word that never passed through `nextQuestion`), the retry slot is burned with no retry enqueued. The next wrong answer sees the cap already hit and blocks further retries.

Not reachable in the current standard flow, but reachable in any host that submits a pre-built result or restores mid-batch state.

**Fix:** Move the increment inside the `if (cached)` block:
```ts
if (canRetryInBatch && canRetryInSession) {
  const cached = state.questionCache.get(id);
  if (cached) {
    nextBatchRetryCounts.set(id, batchRetries + 1);  // only burn slot if actually re-enqueuing
    nextQueue.push(cached);
  }
}
```

## Design / Altitude

### DESIGN-1 ‚Äî `sessionRetryCounts` merge uses `.set()` not `+=`

**File:** `packages/srs-engine-v2/src/engine/adaptive-session.ts:86`

`advanceAdaptiveSession` calls `nextSessionRetryCounts.set(id, count)` where `count` comes from `batchOutput.sessionRetryCounts`. This is correct today because `finishBatch` pre-accumulates the prior session total into the batch output. But the correctness is a hidden dependency: any caller that passes a partial delta (e.g. from stale saved state) will silently reset prior-batch counts. The code would be self-contained and safe if it used additive merge (`+=`), removing the dependency on `finishBatch`'s accumulation.

## Cleanup

### CLEANUP-1 ‚Äî Double shuffle in `assembleBatch`

**File:** `packages/srs-engine-v2/src/engine/assemble-batch.ts:61`

When `shuffle=true`, each `composeWordBatchItems` call already shuffles its output, and then the outer `questions.sort(() => Math.random() - 0.5)` shuffles again. Redundant work. If the inner shuffle is ever removed, the outer one silently compensates, obscuring the ordering contract.

**Fix:** Pass `shuffle: false` to inner thunks and let the single outer shuffle handle ordering (or remove the outer one if per-group ordering is intentional).

**‚úÖ Fixed** ‚Äî `assemble-batch.ts`: inner `composeWordBatchItems` calls now pass `shuffle: false`; single outer sort is the sole shuffle.

### CLEANUP-2 ‚Äî O(n¬≤) tile lookup in `resolveEligibleContexts`

**File:** `packages/srs-engine-v2/demo/learning-io.ts:269`

`allPool.find(w => w.id === id)` inside `.map()` over corpus entries. At 50 sentences √ó 5 words √ó 200 pool items = 50k comparisons per batch. Trivial fix: build a `Map<string, QuizItem>` once before the loop.

**‚úÖ Fixed** ‚Äî `learning-io.ts`: `poolMap = new Map(allPool.map(...))` built once before the `.map()` call; lookups now O(1).
