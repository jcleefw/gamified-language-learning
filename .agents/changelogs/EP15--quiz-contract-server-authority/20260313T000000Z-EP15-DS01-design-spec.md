# EP15-DS01: Quiz Contract — Server-Side Answer Authority

**Date**: 20260313T000000Z
**Status**: Draft

## 1. Feature Overview

The server takes ownership of the full quiz loop: it generates choices for each `multiple_choice` question, withholds the correct key from the batch response, and determines correctness when answers are submitted.

Three changes compose the implementation:

1. **`@gll/api-contract`** — `QuizQuestion` gains `choices`, `QuizAnswer.correct` becomes `selectedKey`, `AnswerResultPayload` gains `submittedKey` + `correctKey`, `ErrorCode` gains `INSUFFICIENT_WORD_POOL`.
2. **`POST /api/srs/batch`** — generates 4-choice distractors from the word pool, shuffles the correct answer into a random slot, stores `correctKeys` in the batch registry (server-side only), validates pool size ≥ 4.
3. **`POST /api/srs/answers`** — accepts `selectedKey` per answer, looks up the correct key from the registry, determines `correct` independently, and returns `submittedKey` + `correctKey` in each result. `POST /api/srs/seed` is added, returning `deckId` in its response body.

---

## 2. Core Requirements

| Requirement | Decision | Rationale |
| --- | --- | --- |
| Choice generation scope | `multiple_choice` questions only | `word_block` and `audio` remain untouched per ADR |
| Distractor source | 3 random `targetText` values from seeded pool, excluding correct word | Self-contained — no external data dependency |
| Choice keys | Fixed `a / b / c / d`, shuffled each call | Standard MC layout; deterministic key alphabet simplifies client rendering |
| Correct key storage | Server-side only in `BatchEntry.correctKeys` — never in batch response | Preserves quiz integrity; Postman gets `correctKey` in answers response instead |
| Pool validation | Return `400 INSUFFICIENT_WORD_POOL` if total unique words in pool < 4 | Fail fast; avoids partial-distractor fallback complexity |
| `QuizAnswer` breaking change | `correct: boolean` → `selectedKey: string` | TypeScript enforces the update across the monorepo |
| Seed route | `POST /api/srs/seed` — calls `seedStore()` and returns `{ deckId, wordCount, phase }` | Eliminates console-log dependency for quiz runner (EP16) |
| Error code | `INSUFFICIENT_WORD_POOL` added to `ErrorCode` enum in `@gll/api-contract` | Consistent error taxonomy; allows typed handling |

## 4. Workflows

**`POST /api/srs/batch` — choice generation:**

```
validate deckId
→ pool = [...wordDetails.values()]
→ if pool.length < 4 → 400 INSUFFICIENT_WORD_POOL
→ for each question (multiple_choice only):
    correctText = wordDetails.get(q.wordId).native
    distractors = 3 random entries from pool where native ≠ correctText
    shuffled = shuffle([correctText, ...distractors.map(d => d.native)])
    choices = { a, b, c, d } = shuffled
    correctKey = key whose value === correctText
→ register(batchId, { questions, correctKeys })
→ return BatchPayload (questions include choices; correctKeys NOT in response)
```

**`POST /api/srs/answers` — server-determined correctness:**

```
validate batchId → get BatchEntry
→ for each submitted answer:
    correctKey = entry.correctKeys[answer.wordId]
    correct = answer.selectedKey === correctKey
→ engineAnswers = answers.map(a => { wordId, isCorrect: correct })
→ processAnswers(engineAnswers, wordStates) → updatedStates
→ return updatedWords with { submittedKey, correctKey, correct, masteryCount, phase }
```

**`POST /api/srs/seed`:**

```
seedStore(getThaiConsonantsWordStates(), getThaiConsonantsWordDetails())
→ return { deckId, wordCount: wordStates.length, phase: 'learning' }
```

## Notes

- **`choices` on all question types vs. `multiple_choice` only**: The wire type adds `choices` unconditionally to `QuizQuestion`. For `word_block` and `audio` questions, `choices` will be an empty object `{}` — the runner (EP16) is responsible for skipping the MC interaction for those types. This keeps the type simple and avoids a union.
- **Distractor uniqueness**: Distractors are drawn from `wordDetails` by `targetText` (native script). If multiple words share the same `targetText`, treat them as the same distractor candidate — the shuffle guarantees the correct answer still has a unique slot.
- **`correctKeys` map key**: Use `wordId` (not question index) so answer lookup is O(1) and order-independent.
- **Seed route idempotency**: Calling `POST /api/srs/seed` multiple times replaces state — no guard needed at this stage (no persistence).
- **Breaking change propagation**: After ST01, `apps/server/src/routes/srs.ts` will have TypeScript errors on the `/answers` handler (still reads `a.correct`). These are intentional and resolved in ST03.
