## ⚠️ Refactoring Opportunities

### 1. `router-guards.ts` has no dedicated test

**File**: `apps/srs-demo/src/router-guards.ts`

This is the highest-risk file in the diff — it's a from-scratch reimplementation of the old `navTo` guard logic via a module-singleton seam (`learningSessionSingleton.ts`), not a mechanical extraction. None of the view/page tests exercise `registerNavigationGuard` itself; they all provide a mocked `learningSession`/`reviewSession` directly. Add `router-guards.test.ts` covering: confirm-prompt on Learning↔Review crossing, mid-quiz exit triggers `finishBatchAndTransition`, `skipNextGuard` bypasses exactly once, curation routes always bypass, and the "finalize failed" path blocks navigation and sets `apiError`.

### 2. `assemble-batch.ts` shuffle fix has no test

**File**: `packages/srs-engine-v2/src/engine/assemble-batch.ts`

No test file exists for this module. The comparator → Fisher-Yates fix is a behavior-changing bug fix with nothing asserting the batch is still a valid permutation (same length, same members) after shuffling — nothing would catch a regression back to `.sort(random)` or an off-by-one in the swap.

### 3. Deep-link boot race on `QuizPage.vue` / `ReviewSessionPage.vue`

**File**: `apps/srs-demo/src/views/QuizPage.vue:36-44`, `apps/srs-demo/src/views/ReviewSessionPage.vue:41-49`

These routes are now deep-linkable (a cold load straight to `/learn/quiz/:deckId` or `/review/session?mode=...`), which wasn't previously possible — `'quiz'`/`'review'` were only reachable after the app's synchronous boot completed. On a cold load, App.vue's `bootApp()` sets `configReady` in `onMounted`, but child `onMounted` hooks (these views) fire first, so `initSession()` / `onReview()` can race `configReady` still being false. Both composables fail closed (`apiError` set, no-op) rather than crash, but the user is left on a blank page with a generic error banner and no retry/back path. Recommend blocking/redirecting until `configReady` is true, or retrying the deep-link init once it flips.

### 4. `learningSessionSingleton.ts` throws instead of failing safe

**File**: `apps/srs-demo/src/composables/learningSessionSingleton.ts:22-27`

`getLearningSession()` throws if the router guard's `beforeEach` runs before App.vue's `setup()` registers the singleton. Currently safe only because Vue Router's initial navigation resolves guards via a microtask, so App.vue's synchronous setup wins the race — an implicit, untested assumption about router internals. Consider `if (!bundle) return true` in `registerNavigationGuard` to fail safe (allow navigation, skip the confirm logic) rather than throw unhandled.

### 5. Non-type import in `router.ts`

**File**: `apps/srs-demo/src/router.ts:1`

`import { createRouter, createWebHistory, RouteRecordRaw } from 'vue-router'` — `RouteRecordRaw` is type-only but not marked `import type`, inconsistent with every other new file in this diff. NIT — `import { createRouter, createWebHistory, type RouteRecordRaw } from 'vue-router'`.
