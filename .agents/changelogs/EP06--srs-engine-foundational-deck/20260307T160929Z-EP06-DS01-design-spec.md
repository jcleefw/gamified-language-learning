## 1. Feature Overview

A single module (`foundational.ts`) implements three mechanics that differentiate foundational (high-frequency) words from curated words in the SRS engine:

1. **Active Limit** — Max 3 foundational words active simultaneously (separate from the 8-word curated active window in EP05)
2. **Continuous Wrong Rule** — 3 consecutive wrong answers on a foundational word resets mastery to 0 (Learning phase only; no lapse, no phase change)
3. **Batch Allocation** — 20% of batch slots allocated to foundational words while the pool is active; drops to 5% once the foundational pool is depleted (all active foundational words past mastery threshold)

These mechanics ensure foundational vocabulary follows a tighter learning loop with faster feedback.

---

## 2. Core Requirements

| Requirement | Decision | Rationale |
|-------------|----------|-----------|
| Foundational active limit | 3 words max | Tighter focus for high-frequency words; distinct from curated 8-word window |
| Continuous wrong threshold | 3 consecutive wrong answers | Enough signal to identify persistent gaps; resets mastery before bad habits form |
| Wrong rule effect | Reset `masteryCount` to 0, stay in `learning` phase | Not a lapse — foundational words only live in Learning phase until mastered |
| Consecutive wrong tracking | New field `consecutiveWrongCount` on `WordState` | Isolated counter; resets on correct answer; only applies to foundational words |
| Batch allocation (active pool) | 20% of `batchSize` slots | Sufficient exposure without overwhelming; balances against curated words |
| Batch allocation (depleted pool) | 5% of `batchSize` slots | Minimal maintenance once foundational pool is exhausted |
| Depletion definition | All active foundational words have `masteryCount >= config.masteryThreshold.foundational` | Clear, measurable threshold aligned with existing mastery concept |
| Slot rounding | `Math.round()` for slot counts | Prevents systematic over/under-allocation at small batch sizes |

## 4. User Workflows

### Foundational Active Limit Flow

```
Learner Starts Batch
  ↓
Count foundational words in learning phase
  ├─ < 3 active?
  │   └─ availableSlots = 3 - activeCount
  │       └─ Fill from eligible foundational pool
  │
  └─ ≥ 3 active?
      └─ availableSlots = 0; no new foundational words introduced
```

### Continuous Wrong Rule Flow

```
Foundational Word Answered
  ├─ CORRECT
  │   └─ Reset consecutiveWrongCount = 0
  │       └─ Normal mastery update (increment masteryCount)
  │
  └─ WRONG
      └─ Increment consecutiveWrongCount
          └─ >= 3?
              ├─ YES → Reset masteryCount = 0, reset consecutiveWrongCount = 0
              │         (stay in learning phase)
              └─ NO  → Continue (no mastery change from wrong rule)
```

### Batch Allocation Flow

```
Compose Batch (totalBatchSize from config)
  ↓
Check foundational pool status:
  All active foundational words have masteryCount >= threshold?
  ├─ NO (pool active)
  │   └─ foundationalSlots = Math.round(totalBatchSize × 0.2)
  │
  └─ YES (pool depleted)
      └─ foundationalSlots = Math.round(totalBatchSize × 0.05)
```

## Notes

- **Foundational Active Limit = 3**: Hardcoded per epic scope; not configurable via `SrsConfig` (no config field exists for this). If configurability is needed later, add `foundationalActiveLimit` to `SrsConfig`.
- **Consecutive Wrong Counter vs. `wrongCount`**: `consecutiveWrongCount` is distinct from `wrongCount` — it only tracks unbroken streaks and resets on any correct answer.
- **No Phase Change on Reset**: Foundational words stay in `learning` phase after mastery reset. They don't lapse (lapse is an ANKI/review-phase concept).
- **Depletion = All Active**: "Depleted" means all foundational words passed the mastery threshold — not that the word list is empty. An empty list is treated as depleted (edge case).
- **No Batch Wiring**: `getFoundationalAllocation` returns slot counts only. Integrating into `composeBatch` is EP04/EP07 scope.
