## 1. Feature Overview

Demo scenarios and integration tests validating EP06 foundational deck mechanics. The demo script (`scripts/demo-srs.ts`) gains three new scenarios exercising all EP06 exports. Integration tests verify that words driven through real `updateMastery` progressions interact correctly with `getActiveFoundationalWords`, `applyFoundationalWrongRule`, and `getFoundationalAllocation` across module boundaries.

Three deliverables:

1. **`scripts/demo-srs.ts`** — Scenarios I, J, K: foundational active limit, continuous wrong rule, batch allocation
2. **`foundational-lifecycle.test.ts`** — cross-module lifecycle: `updateMastery` × `getActiveFoundationalWords` × `applyFoundationalWrongRule`
3. **`foundational-allocation-lifecycle.test.ts`** — cross-module lifecycle: `updateMastery` × `getFoundationalAllocation` × `composeBatch`

## 2. Core Requirements

| Requirement      | Decision                                           | Rationale                                                      |

## Notes

- **No CODEMAP update**: `__tests__/` folders are excluded from CODEMAP per RULES.md. Test files must be self-documenting via file-level doc comments.
- **Config parity**: Tests use the same `SrsConfig` as `demo-srs.ts` for reproducibility. Per-test config overrides (e.g., tighter `activeWordLimit`) are created inline like existing integration tests.
- **`updateMastery` does not call `applyFoundationalWrongRule`**: These are separate functions. The integration boundary is that `updateMastery`-produced states are valid input to `applyFoundationalWrongRule`, and the combined workflow produces correct outcomes.
- **Existing coverage preserved**: These tests do not duplicate existing `batch-lifecycle.test.ts` or `active-window-lifecycle.test.ts` scenarios. They focus specifically on EP06 `foundational.ts` cross-module boundaries.
